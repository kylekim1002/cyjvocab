# 관리자 화면 성능 — 적용 내역 & 추가 권장

## 적용된 내용 (코드)

| 영역 | 내용 |
|------|------|
| **학습 관리 목록** | SSR·`GET /api/admin/learning-modules`에서 문항 전체(`items`) 대신 **`_count.items`만** 조회 → 페이로드·DB 부담 감소 |
| **학습 수정** | 목록에서 「수정」 시 **`GET /api/admin/learning-modules/[id]`**로 문항 포함 상세만 로드 |
| **클래스 관리** | `level` / `grade` / `teacher`를 `select`로 필요 필드만 조회 |
| **캠퍼스** | `teachers`를 `id`, `name`만 `select` |
| **프로덕션 로그** | 관리자 페이지 데이터 조회용 `console.log` 제거(오류용 `console.error`는 유지) |
| **관리자 `loading.tsx`** | 하위 경로 전환 시 스켈레톤으로 빈 화면 완화 |
| **사이드바 `Link`** | `prefetch` 명시로 인접 페이지 선로드 유도 |

---

## 추가로 하면 좋은 것 (선택)

| 항목 | 설명 |
|------|------|
| **학생 관리** | `findMany`에 이미 `select`가 있음. 데이터가 매우 많아지면 **페이지네이션·서버 검색** 검토 |
| **성적 조회** | 기간·필터에 따른 **인덱스**·쿼리 계획 점검(관리자 전용이라 트래픽은 상대적으로 적음) |
| **가상 스크롤** | 클래스·학생·학습 목록이 수천 행이 되면 테이블 **가상화**(예: `@tanstack/react-virtual`) |
| **이미지/엑셀** | 대용량 업로드는 **청크 업로드**·진행률 UI |
| **리전** | Vercel·DB를 같은 권역에 두면 관리자·학생 모두 RTT 감소 (`docs/DEPLOYMENT_SCALING.md`) |
